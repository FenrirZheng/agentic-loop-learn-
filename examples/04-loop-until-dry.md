# loop-until-dry / saturation（連續 K 輪找不到新東西才停）

- **族別**：一、迴圈類
- **控制骨架**：`while 連續空輪數 < K: 再找一輪; 有新發現就歸零計數`
- **停止條件**：連續 K 輪都「找不到任何新東西」才收工。

## Prompt 範例

```text
任務：找出下面這段程式碼所有的 bug / edge case。

去重與停止規則：
- 維護一份 [SEEN] 清單（用「檔案:行:問題摘要」當 key）。
- 每一輪 [ROUND k]：只回報「不在 SEEN 裡的新問題」；把新問題加進 SEEN。
- 若某一輪回報 0 個新問題，[DRY COUNT]+1；有新問題則歸零。
- 連續 2 輪 dry（DRY COUNT = 2）才停，輸出 [SATURATED] 與完整 SEEN 清單。
- 每一輪請「換一個切入角度」找：這輪看併發、下輪看邊界值、下輪看錯誤處理、
  下輪看資源洩漏……不要每輪用同一種眼光重掃。

程式碼：
<<<
{{PASTE_CODE}}
>>>
```

## 為什麼這樣做可以

- **簡單的「找 N 個就停」會漏掉長尾**。來源文件點名這點：找 bug、列 edge case、審計這類任務，真正難抓的問題往往稀疏地藏在後面，`count < N` 會在湊滿數字後就停手。用「連續 K 輪無新發現」當停止條件，逼迴圈一直跑到「真的挖不出東西」。
- **維護 SEEN 去重**是關鍵：否則模型每輪重複報同幾個明顯 bug，永遠達不到 dry，或假性 dry。去重要對「所有看過的」比對，而不是只對「已確認的」。
- **每輪換切入角度**（併發→邊界→錯誤處理…）是「多樣性 > 純重複取樣」的落地：換視角比用同一種眼光重掃更能觸到新問題。

## 什麼時候用 / 不要用

- ✅ 未知數量的發現型任務：bug 獵捕、edge case 枚舉、安全審計、需求遺漏盤點。
- ❌ 已知恰好要 N 個答案的任務——那用 budget 就好，別浪費輪數。

## 陷阱

- **K 太小（=1）會提早收工**：一輪碰巧沒找到不代表挖乾了，建議 K≥2。
- **假性 dry**：去重 key 太粗（只用「問題摘要」）會把不同位置的同型 bug 誤判成重複。key 要含位置。
- **長迴圈會 context rot**：跑很多輪時要配 context 管理（剪掉無用歷史），否則越跑越笨。
