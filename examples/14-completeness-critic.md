# Completeness critic（專問「還缺什麼」的收尾 agent）

- **族別**：三、Adversarial / Critique
- **控制骨架**：`主任務完成 → 一個 critic 專問「還缺什麼」→ 它找到的就是下一輪工作`
- **重點**：不是評「做得好不好」，而是掃「有沒有漏」。

## Prompt 範例

```text
（以下是目前對「{{TASK}}」的完整產出：{{OUTPUT}}）

你是完整性稽核員。不要稱讚、不要潤飾既有內容。只做一件事：找出「缺口」。
逐項檢查並回報：
- 哪個「面向 / 情境 / 使用者類型」完全沒被處理到？
- 哪個「宣稱」沒有被驗證，只是斷言？
- 有沒有「該查而沒查的來源 / 該跑而沒跑的檢查 / 該涵蓋而漏掉的 case」？
- 哪個 modality 或角度根本沒被跑過？

輸出一份 [GAPS] 清單，每條標記嚴重度，並寫「補這個缺口的下一個具體動作」。
若真的找不到缺口，須逐項說明「為什麼這幾個常見缺口在這裡不存在」，不准只說「很完整」。
```

## 為什麼這樣做可以

- **「做得好」和「做全了」是兩件事**。一般的 self-refine / judge 傾向優化「已經寫出來的東西」品質，但對「根本沒寫出來的東西」是盲的。Completeness critic 專門盯這個盲區。
- **把缺口變成下一輪的工作項**：它的產出天然是一份 to-do——這讓它能驅動 [loop-until-dry](04-loop-until-dry.md) 型迴圈（critic 找到缺口 → 補 → 再 critic，直到 critic 說沒缺口）。
- **禁止「很完整」空回應、要求逐項說明為何不缺**，逼它認真掃描而不是敷衍蓋章。

## 什麼時候用 / 不要用

- ✅ 怕「漏」的任務：需求盤點、研究報告、測試 case 覆蓋、安全審計、文件完整度。
- ✅ 當作長迴圈的收尾閥門（每輪結束問一次「還缺嗎」）。
- ❌ 怕「錯」而非怕「漏」的任務——那用 [adversarial verify](12-adversarial-verify.md) 把關正確性。

## 陷阱：警惕「缺口」的基準率

若 critic 產出的缺口清單「幾乎每個項目都中」，要回頭檢查是不是判準太鬆（把普遍存在的性質誤當成缺陷）。一個對近乎全體都成立的「缺口」不帶訊號，反而淹沒真正稀有的漏洞。上報前用真實資料量一下這個 flag 的觸發率。
