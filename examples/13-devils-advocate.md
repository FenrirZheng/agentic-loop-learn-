# Devil's advocate / red-team（對自己剛給的建議做一次挑毛病 pass）

- **族別**：三、Adversarial / Critique
- **控制骨架**：`給出建議 → 立刻切換成「專門唱反調」角色，攻擊自己的建議`
- **與 adversarial verify 差別**：verify 是逐條 finding 派多個 skeptic；devil's advocate 是對「整個建議/計畫」做一次紅隊挑刺。

## Prompt 範例

```text
（你剛才給了一個建議 / 計畫 / 方案：{{RECOMMENDATION}}）

現在切換角色：你是被指派來「打掉這個建議」的紅隊。你的獎勵來自找到它會失敗的方式。
逐項回答：
1. 這個建議最可能在什麼情況下出事？給出 2 個具體失敗情境（不是空泛的「可能有風險」）。
2. 它悄悄假設了什麼前提？哪個前提一旦不成立就整個垮？
3. 有沒有更簡單/更便宜/風險更低的替代做法被跳過了？
4. 如果 6 個月後這個決定被證明是錯的，最可能的原因是什麼？

規則：每一條都要具體到「可以拿去驗證」，禁止用「要注意 X」這種無成本的空話。
最後：基於上面的攻擊，給出修正後的建議，或明確說「攻擊沒有動搖原建議，理由是…」。
```

## 為什麼這樣做可以

- **模型對自己的產出有強烈的確認偏誤**，會傾向護航剛講過的話。明確指派一個「獎勵來自找到失敗」的對抗角色，把它從「辯護者」切換成「攻擊者」，才問得出真話。
- **要求「具體到可驗證、禁止空話」**是這招的靈魂——不加這條，紅隊會產出一堆「要注意安全性」「可能有效能問題」這種零成本、無法反駁也無法行動的評論。逼它給「2 個具體失敗情境」讓批判落地。
- 問「6 個月後若錯了最可能的原因」是一種**pre-mortem**：把未來的失敗拉到現在想，比問「有什麼風險」更能戳到真正的盲點。

## 什麼時候用 / 不要用

- ✅ 在你**採納一個建議/計畫之前**做最後把關；架構決策、上線計畫、要對外的方案。
- ✅ 單一產出、想快速壓力測試（比 full debate 輕）。
- ❌ 已有客觀 oracle 能判對錯時——直接驗，別靠嘴上紅隊。

## 陷阱

- **表演式批判**：不設「禁止空話」規則時，紅隊會給一堆看似盡責、實則無成本的泛泛之論，製造「已嚴格審查」的假象。務必要求每條都可驗證。
