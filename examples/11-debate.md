# Debate（兩個 agent 辯論，第三個當裁判）

- **族別**：三、Adversarial / Critique
- **控制骨架**：`agent A 持正方 ↔ agent B 持反方，來回數輪 → judge 裁決`
- **重點**：用對立立場逼出雙方的弱點。

## Prompt 範例

```text
議題：{{CLAIM，例如「這個 API 應該用 REST 而不是 gRPC」}}

角色 A（正方）：全力論證「應該」。給出你最強的 3 個理由 + 證據。
角色 B（反方）：全力論證「不應該」，並「逐條反駁 A 的理由」。
第 2 輪：A 反駁 B、B 再反駁 A（只針對對方最新論點，不要重講舊的）。

裁判（獨立第三方）：
  你沒有立場。針對雙方「實際交鋒的爭點」逐一判定哪方較有理，
  並指出「哪些論點其實雙方都沒證據、只是斷言」。
  最後輸出 VERDICT + 你採信它的關鍵理由；若證據不足以裁決，明說「未定」。
```

## 為什麼這樣做可以

- **對立角色逼出隱藏假設**：讓同一個模型分別扮演正反方，會逼它把「支持某立場的最強論證」和「反對的最強論證」都攤開，比單方向論述更容易暴露薄弱環節。
- **裁判分離**保持中立，避免產出者自己當自己的裁判。

## ⚠️ 現實檢查：debate 常常贏不過 self-consistency

來源文件的重要提醒，別為辯論而辯論：

> 很多 multi-agent debate 框架**贏不過單純的 self-consistency**；增益大多來自「多次取樣 + 多數決」而非「辯論」本身。

要讓辯論真的有用，得**刻意引入不對稱**，否則兩個同模型互噴只是浪費 token：
- **信心加權**：讓較有把握的一方權重更高。
- **oracle-locking**：一方能引用外部事實/測試結果，把爭論釘在 ground truth 上。
- **角色差異**：給雙方不同的知識、不同的預設立場、甚至不同模型。

## 什麼時候用 / 不要用

- ✅ 有真正對立面、且對立能逼出證據的決策（技術選型、風險評估）。
- ✅ 你願意加入上面的「不對稱」設計。
- ❌ 有唯一解的推理題——先試 [self-consistency](06-self-consistency.md)，通常更省更準。
- ❌ 只是想「顯得嚴謹」而讓兩個同設定 agent 對噴——沒有不對稱時增益很低。
