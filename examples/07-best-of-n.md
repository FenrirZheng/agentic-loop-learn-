# Best-of-N / rejection sampling（生 N 個用 judge 挑最好）

- **族別**：二、Generate-then-Select
- **控制骨架**：`生 N 個候選 → 用一個 judge prompt 挑最好的一個`
- **與 self-consistency 差別**：不是投票數眾數，而是用「評審」擇優——適合沒有唯一解、但好壞可判斷的任務。

## Prompt 範例

生成階段（N 個候選，刻意要多樣）：

```text
為這個需求寫一句 CTA 文案：{{BRIEF}}
請一次給我 5 個「風格明顯不同」的版本（理性/情緒/幽默/急迫/好奇），
標記 A–E。不要 5 個都是同一調性的換字。
```

挑選階段（judge，另一次呼叫）：

```text
你是文案評審。針對候選 A–E，用固定量表逐一打分：
- 清晰（0–3）｜行動誘因強度（0–3）｜與品牌調性契合（0–2）｜獨特性（0–2）
先各自打分並寫「一句話理由」，再輸出：
WINNER: <字母> ── 並說明它為何勝過第二名。
```

## 為什麼這樣做可以

- **生成與挑選分工**：生成階段要的是「發散/多樣」（溫度高、明確要求風格拉開），挑選階段要的是「穩定的判斷」（固定量表）。把兩者混在一次呼叫裡，模型會傾向給一個「安全的平均解」，反而拿不到極值。
- **多樣性是關鍵**：若 5 個候選其實大同小異，挑最好的意義不大。明確指定「5 種不同調性」保證候選張開了解空間——這比對同一個最強設定狂 sample 更有效（多樣性 > 純規模）。
- **相對判斷比絕對判斷容易**：讓 judge 說明「為何勝過第二名」，比要它給一個絕對分數更可靠。

## 什麼時候用 / 不要用

- ✅ **解空間寬**的生成任務：文案、命名、API 設計、重構方案、slogan。
- ✅ 有 judge 能分辨好壞，但沒有唯一正解。
- ❌ 有唯一解的推理題——用 [self-consistency](06-self-consistency.md) 投票更省更準。

## 進階與陷阱

- judge 有位置偏誤（偏好先出現/較長的候選）——把候選匿名化、順序隨機化可緩解。
- 絕對評分難、相對比較容易時，升級成 [tournament 兩兩 PK](08-tournament.md)。
- 極端規模版本是 [AlphaCode 式](frontier/32-alphacode-style.md)：生上百萬個候選，用**客觀 oracle**（測試）過濾 99%，再分群去重——挑選靠 oracle 比靠模型自評準得多。
