# Judge panel（多角度評審各打分再綜合）

- **族別**：二、Generate-then-Select
- **控制骨架**：`同一候選 → 多個「不同角度」的評審各自打分 → 綜合`
- **重點**：用多個專門視角取代單一「全能評審」。

## Prompt 範例

三個獨立評審，各自只管一個面向：

```text
[評審 1 · 正確性]
只從「正確性 / 是否會出 bug / 邊界條件」評這份程式碼，忽略風格。
給 0–10 分 + 最多 3 個具體問題（附行號）。

[評審 2 · 安全性]
只從「安全性 / 注入 / 權限 / 機敏資料外洩」評這份程式碼。
給 0–10 分 + 具體風險點。

[評審 3 · 可讀性]
只從「可讀性 / 命名 / 結構 / 未來維護者的認知負荷」評。
給 0–10 分 + 具體改善建議。
```

綜合（aggregator，另一次呼叫）：

```text
以下是三位評審對同一份程式碼的評語與分數：
{{three_reviews}}
請綜合成一份「行動清單」：
- 依「嚴重度 × 出現在幾個評審口中」排序。
- 同一問題被多位評審提到 → 提到最前面。
- 明確分「Must fix / Nice to have」。
不要只是把三份評語貼在一起——要合併同類、去掉重複、給出優先序。
```

## 為什麼這樣做可以

- **單一全能評審會顧此失彼**：要一個 prompt 同時盯正確性、安全、可讀性，它的注意力被稀釋，容易只抓到最顯眼的一類問題。拆成專職評審，每個都在自己的面向上更深、更敏銳。
- **異質視角 > 同質重複**：三個角度不同的評審，比同一個評審跑三次更能覆蓋不同失效模式（呼應「多樣性 > 純重複取樣」）。
- **aggregator 綜合 > 單純挑一個**：來源文件明確指出，用一個彙整者「重寫綜合」比只用一個 ranker「挑一個」效果好——因為綜合能把各評審的最佳觀點嫁接在一起。

## 什麼時候用 / 不要用

- ✅ 評價標準本身是多維的：code review、設計評審、寫作（內容/結構/文法分開評）。
- ✅ 想要可解釋、可稽核的評分（每個面向分數獨立可查）。
- ❌ 單維度、有客觀 oracle 的任務——直接用 oracle。

## 進階

- 讓每個評審先寫一段推理再判（generative verifier）比只吐分數準。
- 給每個評審一份明確 rubric（見 [rubric judge](frontier/33-rubric-based-judge.md)）能進一步把主觀變半客觀。
