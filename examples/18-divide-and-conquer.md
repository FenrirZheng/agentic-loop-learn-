# Divide-and-conquer / 遞迴（太大就遞迴拆解再合併）

- **族別**：四、Decomposition
- **控制骨架**：`if 夠小: 直接解 else: 拆成子問題 → 各自遞迴 → 合併子解`
- **經典例**：summarize 超長文 → 分段 summary → summary of summaries。

## Prompt 範例

遞迴摘要（樹狀合併）：

```text
你在做遞迴摘要。規則：

BASE CASE：若給你的文字 ≤ 2000 字，直接輸出「不超過 5 點」的摘要。

RECURSIVE CASE：若超過 2000 字，
1. 把它切成數段（每段 ≤ 2000 字，盡量在自然段落邊界切）。
2. 對「每一段」各自產出 5 點摘要。
3. 把所有段的摘要「串起來」當成新的一份文字，再對它套用同一規則
   （若這份合併摘要仍 > 2000 字，就再遞迴一層）。

停止：直到最終摘要 ≤ 2000 字且 ≤ 5 點。
每一層請標記 [LEVEL d]，讓我看得出遞迴深度。
```

## 為什麼這樣做可以

- **同一個「拆-解-合」規則遞迴套用**，讓「大到無法一次處理」變成「一定能處理的小塊」。相比一次性 map-reduce 只有一層，遞迴能處理「連 reduce 階段的輸入都還是太大」的情況——摘要的摘要再摘要。
- **每層都在可控大小內工作**，避免超長 context 造成的「中間遺失」與品質衰減。
- 樹狀結構讓資訊**逐層濃縮**，比硬塞全文更能保住重點層級。

## 什麼時候用 / 不要用

- ✅ 輸入規模遠超單次容量、且問題可自相似拆解：超長文摘要、大型 codebase 逐層歸納、巨型 log 分析。
- ❌ 問題不能自相似拆解（子問題和母問題結構不同）——那用 [plan-then-execute](15-plan-then-execute.md) 明確規劃各異質步驟。

## 與 map-reduce 的分界

- [map-reduce](16-map-reduce.md) = 通常一層（切片 → 平行 → 一次合併）。
- divide-and-conquer = 遞迴多層（合併結果若仍太大，再拆再合）。本質是 map-reduce 的遞迴版。

## 陷阱

- **逐層資訊流失**：每層摘要都會丟細節，深層遞迴後可能失真。關鍵數字/實體要在每層明確要求保留。
- **邊界切斷語意**：同 map-reduce，相鄰段落重疊一點可緩解。
