# Graph-of-Thoughts（GoT，思路是任意圖，可合併多條思路）

- **族別**：四之補（decompose 族）
- **控制骨架**：`思路節點可任意連接成圖 → 允許把多條思路「合併」成一個綜效節點`
- **與 tree 差別**：tree 只能分岔往下；graph 允許不同分支**匯流**成一個更強的節點。

## Prompt 範例

```text
用「思路圖」解這題：{{PROBLEM}}

1. [GENERATE NODES] 產出 3–4 條「不同的解題思路節點」，各標 N1、N2…
2. [EXPAND] 對有前景的節點各展開一步（可再生子節點）。
3. [MERGE] 找出「彼此互補」的節點，把它們合併成一個綜效節點：
   明確寫出「合併後的節點吸收了哪幾個節點的哪些優點」。
   （例：N2 的資料結構 + N3 的邊界處理 → N5。）
4. [SCORE & SELECT] 對目前所有節點評分，挑最強的推進或再合併。
5. 直到得到一個明顯優於任何單一思路的節點，輸出它 + 它的合併來源。
```

## 為什麼這樣做可以

- **好點子常分散在不同思路裡**：tree search 每條路各走各的，無法把「N2 的資料結構」和「N3 的邊界處理」結合。GoT 的**合併節點**允許不同分支的優點匯流，產出單一思路達不到的綜效解。
- **圖比樹更彈性**：除了分岔（發散）和合併（聚合），還能形成任意依賴，貼近真實思考中「這個想法讓我回頭改進那個想法」的非線性過程。
- **要求寫出「合併吸收了哪些優點」**逼模型真的做整合，而不是把兩個點子並列了事。

## 什麼時候用 / 不要用

- ✅ 複雜問題，且「不同部分解可以拼裝成更好的整體解」：演算法設計、方案綜合、需要整合多來源的複雜任務。
- ✅ 你觀察到「幾個候選各有所長，最佳解是它們的組合」。
- ❌ 簡單、單線思路就能解——GoT 的編排開銷不值得。
- ❌ 只需並行展開獨立要點（不需合併）→ 用 [Skeleton-of-Thought](26-skeleton-of-thought.md)；只需分岔回溯 → 用 [tree search](../21-tree-search.md)。
