# ReWOO（Reasoning WithOut Observation，先規劃整條藍圖再一次執行）

- **族別**：四之補(2)（prompt 層規劃法）
- **控制骨架**：`Planner 一次規劃好整條推理藍圖（含要呼叫哪些工具）→ Worker 一次性執行所有工具 → Solver 綜合`
- **與 ReAct 對照**：ReAct 逐步 think-act-observe 來回；ReWOO 先想好再一次做完，省 token、少往返。

## Prompt 範例

Planner：

```text
[PLANNER] 針對「{{TASK}}」，先「一次性」規劃完整解題藍圖，不要現在執行。
輸出編號計畫，每步格式：
  #k: <這步要做什麼> = Tool[<工具>](<參數，可引用前面步驟的結果佔位符 #j>)
把所有需要的工具呼叫都先排好；用 #j 佔位符表達「這步用到第 j 步的結果」。
```

Worker + Solver：

```text
[WORKER] 依 Planner 的藍圖，依序執行每個 Tool[...] 並記下各步結果（填回 #k）。
[SOLVER] 拿到所有步驟結果後，一次綜合出最終答案，不再額外呼叫工具。
```

## 為什麼這樣做可以

- **ReAct 的逐步往返很耗 token**：每一步都要把「到目前為止的完整歷史」再餵一次給模型，來回 N 步就重複 N 次。ReWOO 把推理和觀察**解耦**——先一次規劃好整條藍圖（含所有工具呼叫），再一次執行，大幅減少重複的 context 往返。
- **用佔位符 `#j` 表達步驟依賴**，讓 Planner 不需要真的看到中間結果就能規劃全局——這正是「可事先規劃」任務的關鍵：不必邊看邊決定。
- 規劃與執行分離也讓計畫可被檢查、可重用。

## 什麼時候用 / 不要用

- ✅ 任務**可事先規劃**、步驟與工具呼叫大致可預知、不需要看中間結果才知道下一步。
- ✅ 成本敏感、想省 token。
- ❌ 高度互動、下一步強烈依賴上一步的真實觀察（探索性 debug、開放式搜尋）——那是 [ReAct](../19-react.md) 的場子。
