# Universal self-consistency（開放式答案用 LLM 挑「最一致」的）

- **族別**：二之補（generate-then-select 的新招）
- **控制骨架**：`獨立生 N 個答案 → 讓一個 LLM 讀完全部，挑出「與多數最一致」的那個`
- **與 self-consistency 差別**：普通版靠精確比對投票；當答案是開放式、無法精確比對時，改用 LLM 判斷「哪個最一致」。

## Prompt 範例

生成 N 個（獨立、溫度略高），再用一次彙整呼叫：

```text
以下是同一個開放式問題的 N 個獨立回答：
[1] {{answer_1}}
[2] {{answer_2}}
...
[N] {{answer_N}}

它們是自然語言、無法逐字比對。請你：
1. 找出這些回答「在核心結論上的共識」是什麼。
2. 挑出「最能代表這個共識、且最一致」的那一個回答的編號。
3. 若存在明顯分裂成兩派的情況，指出來，並各報一個代表。
輸出：MOST_CONSISTENT: [編號] + 你判斷它最具代表性的理由。
```

## 為什麼這樣做可以

- **普通 self-consistency 的投票需要「能精確比對答案」**（42 vs 40 這種）。但開放式回答（一段解釋、一份設計）沒有可比對的離散答案，沒東西可投票。Universal 版把「數眾數」這件機器做不到的事，交給一個 LLM 用語意判斷「哪個和大家最一致」。
- **一致性仍是有用的正確性代理**：多個獨立回答收斂到的核心結論，比任一單獨回答更可能對——這個直覺在開放式任務依然成立，只是「一致」的判定從字串比對換成語意判斷。
- **要求指出「分裂成兩派」**避免在真有分歧時硬選一個，掩蓋了不確定性。

## 什麼時候用 / 不要用

- ✅ 開放式、自然語言、無唯一標準答案但「有共識可循」的任務：解釋、建議、摘要要點。
- ❌ 有離散可比對答案 → 用普通 [self-consistency](../06-self-consistency.md) 投票，更客觀。
- ❌ 需要融合多方優點而非挑一個 → 用 [MoA](25-mixture-of-agents.md)。
