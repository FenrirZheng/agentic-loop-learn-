# AlphaCode 式（狂生 + oracle 過濾 + 行為分群去重）

- **族別**：二之補(2)（generate-then-select 的極端規模範例）
- **控制骨架**：`生成海量候選 → 用客觀 oracle 過濾掉絕大多數 → 對存活者依「行為」分群去重 → 每群挑一份提交`
- **重點**：把 best-of-N 推到極致的模板——**生成靠取樣要多樣，挑選靠 oracle 要客觀**。

## Prompt / 流程範例（適用於「有測試可跑」的程式題）

```text
階段 1 [GENERATE]：對題目 {{SPEC}}，用較高溫度生成大量（數十~數百）多樣候選解，
   刻意變化演算法思路，不要都是同一種寫法的微調。

階段 2 [ORACLE FILTER]：對每個候選跑「題目附的範例測試」，
   淘汰所有沒通過的（通常砍掉絕大多數）。← 客觀 oracle，不靠模型自評。

階段 3 [CLUSTER]：對存活的候選，用「另外生成的測試輸入」去跑，
   依「輸出行為」把行為相同的歸為一群（去重）。

階段 4 [SELECT]：每個大群挑一份代表，提交少數幾份。
```

## 為什麼這樣做可以

- **用客觀 oracle 過濾 + 用行為分群去重，遠比讓模型自評挑答案準**——這是來源文件點出的關鍵洞察。模型自評選「看起來最好的」很不可靠；改用「真的跑測試」淘汰、用「實際輸出行為」歸類，把主觀判斷降到最低。
- **生成階段要多樣**：候選若都長一樣，海量也沒用。刻意變化演算法思路，才能讓 oracle 有機會撈到那個對的。這是 [until oracle passes](../05-until-oracle-passes.md) 精神放到 parallel 軸上。
- **行為分群**解決「很多候選其實等價」的浪費——按行為去重後，只需在少數代表裡選，提交次數大幅下降。

## 什麼時候用 / 不要用

- ✅ 有便宜的客觀 oracle（測試）、且能大量平行生成的任務：競賽程式、可自動驗證的生成。
- ✅ 提交/驗證次數有限，需要先大量篩再精選。
- ❌ 沒有客觀 oracle、或生成成本極高——海量生成划不來，改用 [best-of-N](../07-best-of-n.md) 小規模 + judge。
