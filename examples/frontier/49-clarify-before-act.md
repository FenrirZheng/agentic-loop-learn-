# Clarify-before-act（動手前的消歧閘：行為不一致 = 該問了）

- **族別**：五之補（互動類——現有各招全部假設「輸入是明確的」，本篇處理「輸入本身就是錯誤來源」）
- **控制骨架**：`偵測歧義（n 份實作行為比對）→ 有歧義 → 對「分歧點」生成具體澄清問題 → 拿到答案再做；無歧義 → 直接做`
- **為什麼需要**：agent 最貴的失敗不是做錯，是**完美地做對了錯的題**。需求歧義是實務上 agent 翻車的第一大來源，但 43 招裡沒有任何一招處理它。

## Prompt 範例

便宜版（一次呼叫的歧義自查）：

```text
需求：{{REQUIREMENT}}

先別動手。列出這個需求的所有合理解讀。
- 若只有一種 → 直接開始做。
- 若有多種且會導致「不同的輸出」→ 對每個分歧點問一個具體問題，
  附上選項（讓提問可以被一個字回答），然後停下等回覆。
禁止問抽象問題（「你想要什麼風格？」不行；
「標題要含日期嗎？A) 含 B) 不含」可以）。
```

ClarifyGPT 版（用行為不一致**客觀偵測**歧義）：

```text
[STEP 1] 根據需求，生成一組測試輸入。
[STEP 2] 獨立生成 3 份實作（fresh context，互不可見）。
[STEP 3，程式] 3 份實作跑同一組測試輸入：
  - 輸出全一致 → 需求足夠明確，直接採用任一份。
  - 某些輸入點上輸出分歧 → 這些點就是歧義的客觀證據。
[STEP 4] 只針對分歧點生成澄清問題：
  「輸入是 {{X}} 時，應該輸出 {{A}} 還是 {{B}}？」
```

## 為什麼這樣做可以

- **行為不一致是歧義的客觀證據**。直接問模型「這需求模糊嗎」不可靠——它要嘛過度自信（不問就做），要嘛過度發問（什麼都問）。「3 份獨立實作在同一輸入上給出不同輸出」則是不可辯駁的訊號：需求沒有把這個 case 釘死。實測 GPT-4 在 MBPP 上 Pass@1 70.96% → 80.80%。
- **問題錨定在分歧點＋附選項**，使用者能秒答——澄清的摩擦成本低到可以常態使用。
- **同積木第三用途**：這是 [06 self-consistency](../06-self-consistency.md) 的積木（sample 多份看散度）反過來用在**輸入端**——散度大不是答案不可靠，是**題目**不可靠。

## 什麼時候用 / 不要用

- ✅ 需求一句話、實作空間很大（「幫我寫個報表」）。
- ✅ 不可逆動作之前（配 [50 dry-run](50-dry-run-premortem.md)：先確認題目對，再模擬後果）。
- ❌ 需求已經極明確——發問只是摩擦。
- ❌ 探索型任務（「做個原型看看」）——做了才知道要什麼，先做再迭代。
- ⚠️ autonomous 環境沒人可問 → 降級策略：選**最保守**的解讀執行，並把「我假設了什麼」顯式寫進交付物。

## 陷阱

- **每次都問**：閘門的意義是「大多數時候不問」；無條件發問 = 體驗差 + 使用者開始盲目回「隨便」。
- **問抽象問題**：得到的回答同樣抽象，歧義沒有消除。問題必須具體到單一分歧點。
- **用感覺偵測歧義**：跳過行為比對、讓模型直接自評「模糊嗎」，就退化回不可靠的自我感覺。
