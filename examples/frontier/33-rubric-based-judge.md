# Rubric-based judge（給裁判明確量表，逐項打分再加權）

- **族別**：三之補(2)（rubric 化的裁判）
- **控制骨架**：`定義明確 rubric（逐項標準）→ 裁判逐項打分 + 理由 → 加權綜合`
- **重點**：把「籠統給個分」升級成「逐項檢查表」——對**沒有硬 oracle 的開放式任務**特別有用。

## Prompt 範例

```text
你是評審。用下面這份 rubric 評估產出，逐項打分，不要憑整體印象給個總分。

RUBRIC（每項附「什麼算滿分 / 什麼算 0 分」的錨點）：
| 項目 | 權重 | 3 分（滿分）錨點 | 0 分錨點 |
|---|---|---|---|
| 正確性 | 0.4 | 所有事實可查證且正確 | 出現明確錯誤 |
| 完整性 | 0.3 | 涵蓋所有必要面向 | 漏掉關鍵面向 |
| 清晰度 | 0.2 | 一次讀懂、結構清楚 | 需反覆讀才懂 |
| 簡潔度 | 0.1 | 無冗贅 | 大量重複/廢話 |

輸出：逐項給分（0–3）+「指到具體位置」的理由 → 加權總分 → 一句總評。
禁止：不附位置理由的分數、或與逐項分數不一致的總評。
```

## 為什麼這樣做可以

- **把主觀品質拆成半客觀的逐項檢查表**：問「這份好不好」得到的是模型的整體印象，飄忽且難以複現。給定 rubric + 每項的「滿分/零分錨點」，評分變得可解釋、跨產出可比、也更難灌水。
- **錨點（anchor）是關鍵**：光有項目名稱不夠，要說明「什麼樣算 3 分、什麼樣算 0 分」，才能把不同評審/不同次呼叫的尺度對齊。
- **要求逐項理由指到位置**避免「分數與評語脫節」的表演式評分。這是 [judge panel](../09-judge-panel.md) 的標準化升級。

## 什麼時候用 / 不要用

- ✅ 開放式、沒有硬 oracle、但品質有明確維度可拆的任務：寫作、設計、答案品質評分、[until threshold](../03-until-threshold.md) 的自評門檻。
- ✅ 需要可稽核、可複現的評分（例如當作訓練/篩選訊號）。
- ❌ 有客觀 oracle → 直接用 oracle，別用 rubric 當代理。
