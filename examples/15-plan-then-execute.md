# Plan-then-execute（先出計畫，再逐項執行）

- **族別**：四、Decomposition
- **控制骨架**：`先產計畫（不動手）→ 逐項執行計畫`
- **重點**：把「規劃」與「執行」分成兩個階段/兩次呼叫。

## Prompt 範例

階段一（只規劃）：

```text
任務：{{TASK}}。這一步「只做計畫，不要動手實作」。
輸出一份編號步驟清單，每步包含：
- 這步做什麼（動詞開頭）
- 完成的驗收條件（怎樣算做完）
- 它依賴哪些前面的步驟
最後自檢：這些步驟湊起來真的能達成任務嗎？有沒有缺口或順序錯亂？
```

階段二（照計畫做，逐項）：

```text
以下是已核可的計畫：{{PLAN}}
現在只執行「步驟 {{k}}」。做完後對照它的驗收條件自評是否達成，
再進到下一步。不要跳步、不要偷偷改計畫；若發現計畫有誤，先停下來說明。
```

## 為什麼這樣做可以

- **規劃和執行搶同一份注意力時，兩邊都會做差**。邊想邊做時，模型常「跳步」——zero-shot 直接動手容易漏掉中間環節。先強制產出計畫，等於讓它把全局想清楚再進細節。
- **計畫是可檢查、可核可的中間產物**：你（或另一個 agent）能在動手前就攔截爛計畫，成本遠低於做到一半才發現方向錯。
- **逐項執行 + 驗收條件**讓每步都有明確「做完了沒」的判準，避免執行階段又糊成一團。

## 什麼時候用 / 不要用

- ✅ 多步驟、有依賴關係、一旦方向錯代價高的任務。
- ✅ 你想在動手前 review 方向。
- ❌ 極簡單的一步任務——先寫計畫是多餘開銷。
- ❌ 需要「邊看環境邊決定」的互動任務——那是 [ReAct](19-react.md) 的場子。

## 與鄰近招數的分界

- 計畫**由你寫死路徑** = workflow；計畫**由模型邊做邊定** = agent。能用 workflow 就別上 agent（可預測、可測、便宜）。
- 想「先一次規劃好整條含工具呼叫的藍圖再一次執行」→ [ReWOO](frontier/28-rewoo.md)。
- 難題想「由易到難」拆 → [Least-to-Most](frontier/29-least-to-most.md)；治「跳步」→ [Plan-and-Solve](frontier/30-plan-and-solve.md)。
