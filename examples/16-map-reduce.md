# Map-reduce / fan-out-fan-in（切片 → 平行處理 → 合併）

- **族別**：四、Decomposition
- **控制骨架**：`把輸入切成多片 → 每片獨立平行處理（map）→ 合併結果（reduce）`
- **重點**：獨立子任務平行跑，再匯總。

## Prompt 範例

Map（對每一片獨立呼叫，可平行）：

```text
[MAP · 片段 {{i}}/{{n}}]
以下是一份大文件的第 {{i}} 段。只針對「這一段」做事，不要臆測其他段的內容：
{{chunk_i}}
輸出：這段的 3 個重點 + 任何出現的數字/日期/人名（結構化條列）。
```

Reduce（拿到所有片段結果後一次呼叫）：

```text
[REDUCE]
以下是同一份文件所有段落的重點清單：
{{all_map_outputs}}
請合併成一份不重複、有結構的總摘要：
- 合併同類重點、去掉跨段重複。
- 若各段有衝突的數字/事實，明確標出衝突而不是隨便選一個。
- 保留所有關鍵數字/日期/人名。
```

## 為什麼這樣做可以

- **突破單次 context 上限、且天然平行**：一份塞不進 context 或塞進去也會「中間遺失」的大輸入，切片後每片小而聚焦，模型對每片的注意力都是滿的，品質更均勻。
- **map 階段彼此獨立**是關鍵——每個分片 agent 不需要（也不該）知道其他分片內容，這讓它們能平行跑、互不汙染，wall-clock 只等最慢那片。
- **reduce 專責合併衝突與去重**：把「跨片整合」這件難事集中在一個有全局視野的步驟做，而不是奢望每個 map 分片自己協調。

## 什麼時候用 / 不要用

- ✅ 大輸入可以「按片獨立處理」：長文摘要、大量檔案各自分析、批次分類、跨多來源抽取。
- ✅ 想用平行度換 wall-clock。
- ❌ 分片之間有強依賴、必須循序（片 B 要先知道片 A 結論）——那用 [pipeline](17-pipeline.md) 或 [Least-to-Most](frontier/29-least-to-most.md)。

## 拓撲對照：blackboard（共享黑板）

map-reduce 的協調是「切片時一次決定」；當**子任務間需要看到彼此的中間進展**（誰先解出一塊、其他人接著用）時，第三種多 agent 拓撲是 **blackboard**：所有 agent 共讀共寫一塊黑板（共享檔案 / 狀態文件），誰有進展誰寫上去，其他 agent 據此決定下一步；沒有中央 orchestrator 指派。適合「解的結構事先不知道、要靠拼圖湧現」的問題（破譯、跨域診斷）。代價是黑板本身會腐化——需要一個 curator 定期整理（見 [43 context curation](frontier/43-context-curation.md)），且併發寫入要有規約。事先能切乾淨就用 map-reduce，別為了時髦上 blackboard。

## 陷阱

- **切片邊界會切斷語意**：跨片的因果/指代會被切開。可讓相鄰片重疊一小段，或在 reduce 特別留意跨片關係。
- **靜默截斷**：若因成本只處理了前 N 片，務必 log 出「丟了哪些」，別讓「看起來全處理了」蓋過事實。
