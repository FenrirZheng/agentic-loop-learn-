# Tournament / bracket（候選兩兩 PK 晉級）

- **族別**：二、Generate-then-Select
- **控制骨架**：`候選配對 → 每對比較挑贏家 → 贏家再配對 → 直到剩一個`
- **適用**：「絕對評分很難，但相對比較容易」的時候。

## Prompt 範例

單場對決 prompt（重複套用到每一對）：

```text
以下是兩個候選方案，針對同一需求：{{BRIEF}}

[A]
{{candidate_A}}

[B]
{{candidate_B}}

比較規則：
- 只做「A vs B 哪個更好」的相對判斷，不要給絕對分數。
- 依序比三個面向：正確性 → 可維護性 → 簡潔度；哪個面向先分出高下就以它為準。
- 輸出：WINNER: A|B ── 一句話講清楚「決勝的那個面向」是什麼。
若真的難分軒輊，選「風險較低 / 較不會出事」的那個。
```

外層由你（或程式）安排賽制：8 進 4、4 進 2、2 進 1，每場丟一次上面的 prompt。

## 為什麼這樣做可以

- **相對比較的認知負荷遠低於絕對評分**。要 LLM 說「這份設計值幾分（滿分 10）」，分數會漂移、不穩、跨候選不可比；但問「A 和 B 誰好」，它幾乎每次都能給出穩定且有理由的答案。Tournament 把「難的絕對評分」全部化簡成「容易的兩兩比較」。
- **賽制天然做了裁剪**：不用對所有候選給出全序，只要每場分出勝負，log N 輪就能選出冠軍，比讓 judge 一次讀 N 個候選排全序更穩（一次讀太多，判斷會糊）。
- 決勝規則「哪個面向先分高下就以它為準」給了一個**明確的比較優先序**，避免模型在勢均力敵時擺爛選「都不錯」。

## 什麼時候用 / 不要用

- ✅ 候選多、且絕對品質難量化（設計方案、文案、架構取捨）。
- ❌ 候選有客觀分數可算（跑得過幾個測試）——直接用分數排序，別浪費 PK 場次。
- ⚠️ N 很大時 PK 場次 = N-1，成本上升；可先用便宜的粗篩砍掉明顯差的再進賽。

## 陷阱

- **不遞移性**：A>B、B>C 不保證 A>C。若很在意，可對冠亞軍多跑幾場、或改用 [judge panel](09-judge-panel.md) 綜合多角度。
